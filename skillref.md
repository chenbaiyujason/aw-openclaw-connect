# AW OpenClaw Connect Skill

## 目的
通过 activitywatch，获取主人一天的活跃记录，24 小时的记录一般可以用 1～3 万 token 表示。
这个仓库的正式入口是 `aw-connect` CLI。
- 统一通过正式 CLI 调用
- `query` 和 `export` 使用同一套过滤条件
- `query` 输出到标准输出
- `export` 把同样的结果写入文件
- watcher family 是稳定语义层，上游 bucket 结构变化时优先做兼容归类

## 安装

在项目根目录执行：

```bash
python -m pip install -e .
```

安装后可直接使用：

```bash
aw-connect --help
```

## 核心命令

### 1. 查看设备

```bash
aw-connect devices
```

输出逻辑设备到 watcher family 的映射。

### 2. 查看 watcher

```bash
aw-connect watchers
```

```bash
aw-connect watchers --device macbook
```

## 查询与导出

`query` 和 `export` 的过滤条件完全一致，结果语义也一致。

区别只有：

- `query`：打印到标准输出
- `export`：写入文件

### 默认建议

当 agent 只是想补看“用户最近在做什么”时，默认优先只传时间条件。

常用例子：

```bash
aw-connect query --minutes 15
```

```bash
aw-connect export --minutes 60
```

### 完整过滤

如需更细粒度条件，可以继续叠加设备、watcher、绝对时间范围：

```bash
aw-connect query --start 2026-03-06T10:00:00Z --end 2026-03-06T11:00:00Z --device macbook --watcher window
```

```bash
aw-connect export --start 2026-03-06T10:00:00Z --end 2026-03-06T11:00:00Z --device macbook --watcher web --output logs/macbook-web.csv
```

## 什么时候应该用这个 Skill

默认在下面这些场景主动考虑使用：

- 主人和你离开超过 30 分钟后再次对话
- 主人主动问“我刚刚 / 某段时间做了什么”
- 你需要补齐一段你并不在场的上下文
- 你正在处理某件事，想知道主人离开期间有没有继续推进同一件事
- 你需要把近期活动落回记忆文件，而当前记忆明显缺失

如果你是 OpenClaw 风格的长期陪伴 agent，建议把这条规则当成默认工作流：

- 主人回来后，先检查记忆里有没有这段离开期的记录
- 如果记忆里已经有，不要重复抓同一段 log
- 如果记忆里没有，就主动获取“上一次离开到这次回来”的 log
- 获取后按后文的规则理解、清洗并写回对应日期的记忆文件

如果主人直接要求你了解某个时间段做了什么，也按同样流程处理：

- 明确时间范围
- 获取这段时间的 log
- 清洗并总结
- 如有长期记忆职责，再落回记忆后继续对话

## 过滤规则

- `--device` 按逻辑设备名过滤
- `--watcher` 按 watcher family 过滤，例如 `window`、`web`、`vscode`
- watcher family 过滤会自动包含对应的 synced bucket
- agent 不需要关心底层 bucket 名里的 `synced-from-*`
- 对 `aw-watcher-cursor` 这类已拆分为多轨道的来源，仍然统一兼容到 `vscode` family
- 当前 `vscode` family 会兼容文件活动、Agent 生命周期、Git commit 归档等多类事件
- 对文件活动事件，会保留 `activityKind`（如 `dwell` / `edit`），避免把停留阅读和实际编辑混成同一段

## 输出格式

`query` 默认输出、以及 `export` 写入的文件，都是 agent 友好的 CSV：

1. 头部是 `#` 开头的 meta 行
2. 正文列为 `o(s),d(s),dev,w,sub,items`

重点字段：

- `# start` / `# end`：查询时间范围
- `# afk`：是否应用 AFK 清洗
- `# dm`：设备短码映射
- `# ws`：结果中出现的 watcher family
- `# ues`：跨设备并集后的有效时长秒数，不是把每台设备活跃时长直接相加
- `o(s)`：相对开始时间的偏移秒
- `d(s)`：该事件段实际时长
- `dev`：该事件段的来源设备短码，不代表这段时间只有这台设备在活跃
- `w`：watcher family
- `sub`：主题，例如应用名或域名
- `items`：聚合后的详细内容

其中对 `vscode` 结果有额外约定：

- `w` 仍然是稳定的 `vscode` family，而不是暴露底层 bucket 名
- `sub` 通常是项目标识；对文件活动，优先保留项目根路径或明确项目名
- `items` 对文件活动会尽量写成 `文件路径 [activityKind]:时长`
- 当 `sub` 是明确项目路径时，`items` 里的文件路径会尽量压成相对路径，例如 `src/main.ts [edit]:12.3`
- 当 `sub` 是 `unknown` 或项目根不明确时，`items` 会继续保留绝对路径，避免错误截断

## 拿到 log CSV 后怎么处理

拿到 `query` 输出或 `export` 生成的 CSV 后，不要直接把原始行当成记忆。

推荐流程是：

1. 先读 meta，确认时间范围、是否做过 AFK 清洗、涉及哪些设备和 watcher
2. 再按时间顺序理解事件段，而不是逐行机械复述
3. 把相邻且语义相近的事件整理成自然语言片段
4. 再把这些片段写回对应日期的记忆文件

### 清洗成自然语义的目标

目标不是记“系统记录到了什么字段”，而是结合你过往的记忆和上下文记成：

- 在什么时间
- 主人用什么设备
- 为了什么目的
- 做了什么
- 做到什么阶段

更自然的表达应该接近：

- `14:10-14:35，主人在 A 设备上主要在 Cursor 里修改 aw-watcher-cursor，期间查看过 GitHub/文档，像是在推进 ActivityWatch 相关改动。这和主人昨天让我调研的同步时间项目是一件事。`
- `19:40-19:55，没有明显活动。`
- `21:05-21:30，主人切到 B 设备后继续处理前面同一件事，并出现了 commit，说明已经完成并提交了一轮修改。`

### 如何做事件清洗

- 先按 `commit > agent > edit > dwell > web/window` 的优先级理解主线
- `web` 和 `window` 大多作为上下文补充，不要喧宾夺主
- 看到 `localhost:5600` 这类本地 AW 页面时，通常只记成“主人查看时间记录 / 本地统计”，不要误写成任务本身
- 如果多条事件指向同一目的，可以合并成一小段自然语义
- 如果一段时间确实没有有效活动，也要显式写出“这段时间没有明显事情发生”

### 多设备综合判断

当同一时间范围里同时出现多台设备时，不要把它们当成三个人的三条独立流水。

默认要把它们先看成“同一个人此刻有哪些并行活动”，再判断：

- 哪台设备承载主线任务
- 哪台设备是在补资料、对照信息或辅助推进
- 哪台设备只是背景噪音、娱乐、挂机或无关占用

先记住两个底线：

- `# ues` 是跨设备并集时长，所以多设备重叠时间不能直接累加成人类活动总时长
- 单行里的 `dev` 只说明事件从哪台设备来，不等于这段时间只有这台设备重要

推荐按下面顺序综合判断：

1. 先按时间把不同设备的重叠片段对齐，找出同一时段里有哪些设备同时活跃
2. 在每个重叠片段里，先看有没有 `commit`、`edit`、强相关 `agent` 这类高价值信号
3. 把高价值信号所在的设备优先视为主线设备
4. 再看其他设备是在补资料、聊天沟通、查文档、监控结果，还是单纯娱乐/挂机
5. 最后写成“主线 + 辅助 + 背景”的自然语义，而不是按设备逐条平铺

多设备同时活跃时，默认优先级仍然是：

- `commit` 高于 `agent`
- `agent` 高于 `edit`
- `edit` 高于 `dwell`
- `dwell` 高于普通 `web` / `window`
- 与当前任务无关的娱乐、挂机、背景应用，优先级最低

具体怎么判断：

- 如果 A 设备在 `vscode [edit]`，B 设备在看官方文档或教程页，优先理解成“主人一边写代码，一边查资料”
- 如果 A 设备在 `vscode [edit]`，B 设备只是聊天页、音乐页、统计页，通常把 B 当背景上下文，不要抬到主线
- 如果 A 设备在写代码，B 设备在看教程，C 设备只是挂机游戏，主线通常仍然是代码任务；教程页是辅助证据；挂机游戏只在确实影响注意力或切走很久时才值得一提
- 如果两台设备都出现强信号，例如 A 设备 `edit`、B 设备也有强相关 `agent` 或 `commit`，可以写成“主人在跨设备并行推进同一件事”或“同时处理两条任务”，不要硬编成单线程叙事
- 如果多台设备主题一致，例如一台改代码、一台看同项目 PR / issue / 文档，通常应合并理解成同一任务
- 如果多台设备主题明显无关，例如 A 在项目开发，B 在购物，C 在游戏，就要承认主人在分心或并行做多件事，不要强行说它们都服务于同一个目标

写回记忆时，优先写“人”的状态，不要写“设备列表”：

- 好的写法：`20:10-20:40，主人主线在 A 设备上改 aw-openclaw-connect，同时用 B 设备查 ActivityWatch 文档；C 设备有挂机游戏，但看起来不是这一轮任务的核心。`
- 不好的写法：`20:10-20:40，A 设备 edit，B 设备 web，C 设备 window。`

### 空白时间也要记

如果清洗后发现一个时间跨度内没有有效活动，不要省略。

应该显式记录类似：

- `16:20-17:10，没有明显活动。`

这样后续 agent 才能知道：

- 这段时间是已经补看过的
- 不是漏记
- 不是还没抓 log

## Watcher 速查

下面这部分不是 CLI 语法，而是“看到结果后该怎么读”。

### `afk`

这类记录表示设备是否处于活跃状态，主要常见字段是：

- `status`：常见值是 `not-afk` 或 `afk`

它的作用不是告诉你主人具体在做什么，而是告诉你这段时间是否应当算有效活动时间。

怎么用：

- `afk` 更像清洗边界，不像行为内容
- 当结果已经启用了 AFK 清洗时，读取 agent 一般不用再把它当成行为线索
- 它最适合回答“这段时间主人是不是还在电脑前”

### `window`

这类记录表示当前前台应用和窗口标题，常见字段通常是：

- `app`：应用名，例如 `Cursor`、`Google Chrome`
- `title`：窗口标题

它通常表示“主人当前把哪个应用放在最前面”，是上下文线索，不一定是成果本身。

怎么用：

- `window` 适合判断主人大致在用什么应用
- 对 `Cursor` / `Code` / 浏览器这类壳子窗口，要优先把它当补充上下文
- 如果同时有更具体的 `vscode` 或 `web` 记录，优先相信更具体的记录
- 单看 `window`，不要轻易推断主人具体完成了什么

### `web`

这类记录表示浏览器里的页面活动，常见字段通常是：

- `url`：完整网页地址
- `title`：网页标题

导出后通常会表现为：

- `sub`：域名或站点，例如 `github.com`
- `items`：更具体的 URL

它的语义通常是“主人在看哪个网页或网页应用”，更偏资料、控制台、文档、聊天或后台页面。

怎么用：

- `web` 很适合补上下文，比如主人在看文档、PR、issue、搜索结果，还是某个管理后台
- `web` 本身不等于工作成果，更多是过程和背景
- `localhost:5600` / `127.0.0.1:5600` 这类本地 ActivityWatch UI 页面，通常表示主人在看自己的时间记录或本地统计，不是任务本身的有意义内容
- 如果网页只是本地面板、空白页、设置页或监控页，一般不要把它当成“主人真正完成的事情”

### `vscode`

这类记录统一兼容来自 `aw-watcher-cursor` 的多种轨道，当前常见会混合这些语义：

- 文件活动 `fileActivity`
- 文件活动里的 `activityKind`
- Agent 生命周期事件
- Git commit 归档事件

在导出结果里，常见能看到的字段线索包括：

- `sub`：通常是项目标识，例如项目根路径
- `items`：通常更具体；对文件活动，常见是 `相对路径 [activityKind]`，项目未知时则保留绝对路径

怎么读：

- `fileActivity [dwell]`：表示主人在文件里停留、阅读、思考、对照，不等于真的改了代码
- `fileActivity [edit]`：表示主人明确在改代码，可信度高于 `dwell`
- `agent` 相关事件：表示主人在用 Cursor agent 处理任务，它是过程信号，说明主人在尝试解释、修复、生成、应用或拒绝某个 agent 过程
- `commit`：语义上最重要，代表主人明确提交了一轮修改，通常最接近“完成了一个阶段性事情”

读 `vscode` 时的注意点：

- `dwell` 可以理解成“在看”，不要直接当成“在改”
- `edit` 可以理解成“在动手写”
- `agent` 可以理解成“在借助 agent 推进”
- `commit` 可以理解成“明确完成并提交了一轮结果”
- 当前导出对 `commit` 的呈现不一定总是非常显眼，所以看到可疑的 `vscode` 结果时，也要结合前后的 `edit`、`agent`、`web`、`window` 一起判断

## 如何判断主人做了什么

默认建议按下面顺序读结果：

1. 先找 `commit`
2. 再找强相关 `agent`
3. 再找 `edit`
4. 再看 `dwell`
5. 最后才用 `web` / `window` 补上下文

可以直接记成一条总原则：

- 结果信号强于过程信号
- 过程信号强于背景信号

高价值信号通常包括：

- 明确的 `commit`
- 明确的 `edit`
- 和当前代码上下文强相关的 `agent` 操作

低价值或背景信号通常包括：

- 只显示应用名的 `window`
- 本地统计面板、设置页、监控页这类 `web`
- 只有 `dwell` 但没有后续动作的停留记录

不要过度解读：

- `web` 不等于工作产出
- `window` 不等于实际内容
- `dwell` 不等于修改
- `agent` 不等于最终完成
- `commit` 最接近完成信号，但仍建议结合前后上下文判断
- 当上游 watcher 输出不够显眼或字段被压缩时，不要因为单条记录信息少，就武断下结论

## 如何落回记忆文件

把 log 理解完后，应该写回记忆，而不是只在当前上下文里“脑补记住”。

默认写回目标：

- 当天或对应日期的 `memory/YYYY-MM-DD.md`
- 如果是日终汇总，再补到长期记忆文件，例如 `MEMORY.md`
- 原始或清洗后的 CSV log 文件，也要保存在 `memory/` 目录下面，方便后续 agent 回查

### CSV log 文件的默认存储位置

为了让后续 agent 能在需要时回看更细粒度内容，导出的 CSV 不要散落在任意地方。

默认建议存到：

- `memory/activitywatch-logs/YYYY-MM-DD/`

文件名建议带上时间范围和是否清洗，例如：

- `memory/activitywatch-logs/2026-03-06/2026-03-06T14-00_to_2026-03-06T16-00_cleaned.csv`
- `memory/activitywatch-logs/2026-03-06/2026-03-06T18-30_to_2026-03-06T21-00_cleaned.csv`

如果同一天补抓多次：

- 不要覆盖旧文件
- 让文件名保留时间范围
- 后续记忆可以引用多个 CSV 文件

### 在当日 memory 文件顶部登记 CSV 文件名

每次有新的 CSV log 落回时，都要在对应日期的 `memory/YYYY-MM-DD.md` 最上方维护一个简短索引块。

推荐格式：

```md
## ActivityWatch Logs

- `2026-03-06T14-00_to_2026-03-06T16-00_cleaned.csv`
- `2026-03-06T18-30_to_2026-03-06T21-00_cleaned.csv`
```

这样做的目的不是给人看排版，而是方便其他 agent 在后续场景里快速知道：

- 这一天哪些时间段已经导出过 log
- 如果要追更细节，应该去翻哪一个 CSV
- 当只想知道某个重要网页 URL、某次切换的细节、某段时间里具体看了什么资料时，可以直接定位到对应文件

这个索引块应该保持可更新：

- 新导出一个 CSV，就把文件名补进去
- 如果同一时间段换了更完整版本，也要更新索引
- 不要把这块埋在文件中间，尽量放在日期记忆文件顶部

写回时建议遵守这些规则：

- 先写时间范围，再写自然语义结论
- 一次落回只补“之前缺失”的时间段，不要反复全量重写
- 如果同一天后续又拿到了新设备的新记录，可以回写更新已有条目
- 新信息优先补充，不要把之前已确认的内容删没

### 可更新的回写规则

记忆落回不是一次性的。

例如：

- 白天先拿到 A 设备的 log，发现 `18:00-20:00` 没有明显活动
- 晚上 B 设备后来同步出 `19:10-19:40` 的真实记录

这时应该回到同一天的记忆文件，把原先“没有明显活动”的条目更新成更准确的版本，而不是让旧结论一直留着。

同时也要一起更新顶部的 CSV 索引块，让后续 agent 知道这一天已经有哪些补抓文件。

### 只优先补最近 24 小时

考虑到主人可能是后来才安装 watcher，默认优先补全最近 24 小时内缺失的 log。

对更早的数据：

- 如果主人明确要求，再去补
- 如果只是例行记忆回填，一般不用主动补太早的历史

## 如何把 log 和现有上下文结合

这个 Skill 的真正价值，不只是“知道主人点了什么”，而是把 log 和你已有的上下文、历史记忆、当前任务串起来。

你应该主动做这些判断：

- 你现在正在做的事，和 log 里出现的活动是不是同一件事
- 记忆里主人之前提到过的事，和 log 里后续动作是不是连得上
- 主人离开前让你做 A，离开期间主人自己是不是继续把 A 推进到了 B

如果能判断出相关性，就不要只记“主人做了 B”。

而应该更新成更完整的状态理解，例如：

- 主人先和你讨论了 A
- 离开期间主人自己继续推进，做到了 B
- 所以主人这次回来时，你应当把当前状态理解成 C，而不是还停留在 A

也就是说，log 不是孤立事实，它应该拿来更新你对“这件事现在推进到了哪里”的判断。

## 主动联想与行为推断

不要过于粗浅地把 log 理解成“某时某刻发生某事”的离散流水账。

更有价值的方式是：

- 看事件前后顺序
- 看切换关系
- 看一段时间内主题怎么递进
- 看网页、代码、agent、commit 之间是不是在互相支撑

目标是尽量推测：

- 主人当时在试图解决什么问题
- 主人先缺什么信息，后来补到了什么信息
- 哪些资料、页面、搜索结果、文档、PR 或 issue 可能启发了主人
- 主人是从阅读走向编辑，还是从卡住走向提交

### 看切换，而不是只看点状事件

很多真实意图是通过切换暴露出来的。

例如：

- 先出现 `web` 文档页，再出现 `vscode [edit]`
- 说明主人大概率是先查资料，再回代码里实现

又例如：

- 先出现 `vscode [dwell]`
- 然后切到某个搜索页、issue、PR、官方文档
- 再回到 `vscode [edit]`

这通常表示主人先在代码里卡住或思考，然后出去找资料，再回来继续推进。

### 从 URL 推测资料来源和启发关系

对 `web` 不要只记录“主人打开了某个网页”，而要试着理解这个网页在当时起了什么作用。

比如：

- 官方文档 URL：可能是在查 API、配置、行为定义
- GitHub PR / issue：可能是在看历史上下文、实现方式、bug 讨论
- 搜索结果页：可能说明主人对问题还没有明确答案，正在探索
- 本地 `localhost:5600` ActivityWatch 页面：通常是在看自己的时间记录，而不是推进任务本身

如果前后关系明显，可以进一步推测：

- 哪个资料页是为了某个代码改动服务
- 哪个页面可能直接触发了后续编辑
- 哪些页面只是路过，哪些页面像是关键参考

### 用递进关系理解主人的心路历程

当一段事件有明显递进时，应该主动尝试总结主人的心路历程。

例如可以这样理解：

- 先阅读代码和停留：主人在理解现状
- 然后查资料 / 看 issue / 看 PR：主人在找线索或确认方向
- 再开始 `edit`：主人开始动手实现
- 最后 `commit`：主人确认这一轮已经完成

也就是说，最终写回记忆时，优先记成这样的自然语义：

- 主人先在想什么
- 后来去看了什么资料
- 这些资料可能怎样启发了后续动作
- 最后推进到了什么阶段

### 主动推测，但不要装作确定

你应该主动联想，但表达上要区分强弱：

- 证据很强时，可以直接说“主人在推进 X”
- 证据中等时，可以说“看起来像是在排查 / 对照 / 查资料 / 为某个改动做准备”
- 证据较弱时，可以说“可能是在……”或“像是在……”

好的目标不是绝对机械，也不是胡乱脑补，而是带着上下文做有依据的推测。

## OpenClaw 风格 agent 的额外约定

如果你是 OpenClaw 这类会维护长期记忆的 agent，除了会话内理解，还要把这套流程接到固定记忆文件里。

### 回来后主动补看

- 当主人和你离开超过 30 分钟后再次对话，先检查记忆里是不是已经有这段时间的记录
- 这里的“记忆”既包括当天的 `memory/YYYY-MM-DD.md`，也包括其他 session 可能已经提前写入的内容
- 如果已经有，就直接用，不重复抓
- 如果没有，就主动获取从上一次离开到这次回来的 log，并落回记忆

### 主人主动要求时

当主人主动说“帮我了解某段时间做了什么”时：

- 优先按主人给的时间范围取 log
- 如果主人只表达了“离开期间”但没给具体时间，就补抓你不知晓的那段缺口

### 每 2 小时一次的检查

如果你有定时器或 heartbeat 机制，建议每 2 小时检查一次是否需要补记忆。

触发时建议这样判断：

- 先判断现在是不是主人的清醒时间
- 默认可按早上 8 点到晚上 12 点理解
- 也可以结合 `USER.md`、已有记忆和主人最近作息自行判断
- 如果不是清醒时间，这次检查直接跳过，不输出任何打扰

如果是在清醒时间：

- 检查最近是否已经有对应时间段的记忆落回
- 如果没有，就补抓缺失时间段的 log
- 把清洗结果落回当天记忆文件

### 每天 12 点的日终汇总

如果这次定时触发落在当天晚上 12 点附近，再额外做一层日终整理：

- 读取当天所有相关记忆
- 结合当天补回的 log 和已有上下文
- 用简单几句话写回长期记忆文件

日终汇总的重点不是流水账，而是：

- 主人这一天主要都在推进什么
- 哪些事情有明确结果
- 哪些事情还停在中间阶段

## 行为约定

- 系统只负责返回清洗后的事件
- 不再提供 `user_view` 之类的二次推断视角
- 用户在做什么、哪些事件应当归并、哪些行为和当前对话有关，由 agent 自己判断
- 当上游 watcher 新增 bucket、拆分轨道或调整事件字段时，优先继续通过现有 watcher family 语义消费结果，不要要求用户先记住新的底层命名
